## 12. 增强骨骼生成步骤明细表 (skeleton_pipeline_step)

每条主任务对应 6（FROM_REFERENCE 路径）或 7（FROM_SCRATCH 路径）条步骤记录，记录每步状态、耗时和中间产物 URL。

### 表结构

| 字段名 | 类型 | 不为空 | 索引 | 默认值 | 说明 |
|--------|------|--------|------|--------|------|
| `id` | BIGINT | ✓ | PK | AUTO_INCREMENT | 主键ID |
| `task_id` | VARCHAR(64) | ✓ | ✓ | - | 关联的主任务 task_id |
| `step_no` | INT | ✓ | - | - | 步骤序号（联合唯一） |
| `step_name` | VARCHAR(100) | ✓ | - | - | 步骤名称，如：图层分解 |
| `step_key` | VARCHAR(50) | ✓ | - | - | 步骤唯一键，如：see_through_decompose |
| `status` | VARCHAR(20) | ✓ | ✓ | PENDING | PENDING/PROCESSING/SUCCESS/FAILED/SKIPPED |
| `progress` | INT | - | - | 0 | 步骤内进度 0-100 |
| `error_message` | TEXT | - | - | - | 步骤错误信息 |
| `output_image_url` | VARCHAR(500) | - | - | - | 该步骤输出的图片URL |
| `output_data_json` | TEXT | - | - | - | 该步骤输出的结构化数据（JSON） |
| `output_file_url` | VARCHAR(500) | - | - | - | 该步骤输出的文件URL |
| `started_at` | TIMESTAMP | - | - | NULL | 步骤开始时间 |
| `completed_at` | TIMESTAMP | - | - | NULL | 步骤完成时间 |
| `duration_ms` | BIGINT | - | - | - | 步骤耗时（毫秒） |
| `created_at` | TIMESTAMP | ✓ | - | CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | ✓ | - | CURRENT_TIMESTAMP | 更新时间 |

### step_key 枚举（两条路径）

**FROM_REFERENCE 路径（6步）**

| step_no | step_key | step_name |
|---------|----------|-----------|
| 1 | `see_through_decompose` | see-through 图层分解 |
| 2 | `sam2_refine` | SAM 2 精修图层 |
| 3 | `bg_remove_per_layer` | 逐层去背 |
| 4 | `keypoint_detect` | 关键点识别 |

## 索引优化策略

### 复合索引

```sql
-- 游戏表
CREATE INDEX idx_game_user_type ON game(user_id, type);
CREATE INDEX idx_game_created_updated ON game(created_at, updated_at);

-- AI任务表
CREATE INDEX idx_ai_task_status_created ON ai_portrait_task(status, created_at);

-- AI生成表
CREATE INDEX idx_ai_generation_user_status ON ai_portrait_generation(user_id, status);
```

### 索引使用建议

1. **避免全表扫描**: 确保WHERE子句中的字段有索引
2. **避免索引失效**: 不要在索引字段上进行函数操作
3. **定期分析**: 使用 `ANALYZE TABLE` 更新表统计信息
4. **监控性能**: 定期检查慢查询日志
