# 数据库视图、触发器和存储过程

## 视图 (Views)

### 1. 游戏统计视图 (v_game_statistics)

用途：提供游戏的统计信息，包括按游戏类型的统计。

```sql
CREATE OR REPLACE VIEW v_game_statistics AS
SELECT
    type,
    COUNT(*) as game_count,
    COUNT(DISTINCT user_id) as user_count,
    MIN(created_at) as first_created,
    MAX(created_at) as last_created,
    AVG(TIMESTAMPDIFF(DAY, created_at, updated_at)) as avg_update_days
FROM game
GROUP BY type;
```

**查询示例：**

```sql
-- 查看所有游戏类型的统计
SELECT * FROM v_game_statistics;

-- 查看特定类型的统计
SELECT * FROM v_game_statistics WHERE type = '2d-strategy';

-- 查看拥有最多游戏的类型
SELECT * FROM v_game_statistics
ORDER BY game_count DESC
LIMIT 1;
```

**列说明：**

| 列名 | 类型 | 说明 |
|------|------|------|
| `type` | VARCHAR(50) | 游戏类型 |
| `game_count` | BIGINT | 该类型游戏总数 |
| `user_count` | BIGINT | 创建该类型游戏的用户总数 |
| `first_created` | TIMESTAMP | 该类型首个游戏的创建时间 |
| `last_created` | TIMESTAMP | 该类型最后创建的游戏时间 |
| `avg_update_days` | DECIMAL | 平均更新间隔天数 |

### 2. AI任务统计视图 (v_ai_task_statistics)

用途：提供AI任务的每日统计信息，按状态分类。

```sql
CREATE OR REPLACE VIEW v_ai_task_statistics AS
SELECT
    DATE(created_at) as task_date,
    status,
    COUNT(*) as task_count,
    AVG(progress) as avg_progress,
    AVG(retry_count) as avg_retries,
    SUM(CASE WHEN last_error IS NOT NULL THEN 1 ELSE 0 END) as error_count
FROM ai_portrait_task
GROUP BY DATE(created_at), status;
```

**查询示例：**

```sql
-- 查看最近7天的AI任务统计
SELECT * FROM v_ai_task_statistics
WHERE task_date >= DATE_SUB(CURDATE(), INTERVAL 7 DAY)
ORDER BY task_date DESC;

-- 查看今天成功任务数
SELECT task_count FROM v_ai_task_statistics
WHERE task_date = CURDATE() AND status = 'SUCCESS';

-- 查看所有失败任务及其错误率
SELECT
    task_date,
    task_count,
    error_count,
    ROUND(error_count / task_count * 100, 2) as error_rate
FROM v_ai_task_statistics
WHERE status = 'FAILED'
ORDER BY task_date DESC;
```

**列说明：**

| 列名 | 类型 | 说明 |
|------|------|------|
| `task_date` | DATE | 任务创建日期 |
| `status` | VARCHAR(20) | 任务状态 |
| `task_count` | BIGINT | 该状态的任务数 |
| `avg_progress` | DECIMAL | 平均进度 |
| `avg_retries` | DECIMAL | 平均重试次数 |
| `error_count` | BIGINT | 包含错误的任务数 |

## 存储过程 (Stored Procedures)

### 1. 清理过期任务存储过程 (sp_cleanup_old_tasks)

用途：删除指定天数前的已完成或已失败任务，用于数据库维护和清理。

```sql
DELIMITER //
CREATE PROCEDURE sp_cleanup_old_tasks(IN days_to_keep INT)
BEGIN
    DECLARE cutoff_date TIMESTAMP;
    SET cutoff_date = DATE_SUB(NOW(), INTERVAL days_to_keep DAY);

    -- 删除过期的成功任务
    DELETE FROM ai_portrait_task
    WHERE status = 'SUCCESS'
      AND completed_at < cutoff_date;

    -- 删除过期的失败任务（已超过最大重试次数）
    DELETE FROM ai_portrait_task
    WHERE status = 'FAILED'
      AND retry_count >= max_retries
      AND updated_at < cutoff_date;

    SELECT ROW_COUNT() as deleted_count;
END //
DELIMITER ;
```

**使用示例：**

```sql
-- 删除30天前的完成任务
CALL sp_cleanup_old_tasks(30);

-- 删除90天前的完成任务
CALL sp_cleanup_old_tasks(90);
```

**参数说明：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `days_to_keep` | INT | 保留的天数 |

**返回值：**

| 列名 | 说明 |
|------|------|
| `deleted_count` | 删除的任务总数 |

**执行逻辑：**

1. 计算截止日期（当前时间 - days_to_keep天）
2. 删除所有已完成且完成时间早于截止日期的任务
3. 删除所有已失败、重试次数已超过上限、且更新时间早于截止日期的任务
4. 返回删除的总行数

## 触发器 (Triggers)

### 触发器命名规范

- 格式: `tr_{table}_{event}_{time}`
- 例: `tr_game_after_insert` 表示game表INSERT后触发

---

### 1. 游戏表插入触发器 (tr_game_after_insert)

用途：在新建游戏时自动记录操作日志。

```sql
DELIMITER //
CREATE TRIGGER tr_game_after_insert
AFTER INSERT ON game
FOR EACH ROW
BEGIN
    INSERT INTO operation_log (user_id, operation_type, operation_target, operation_detail)
    VALUES (NEW.user_id, 'GAME_CREATE', 'game', CONCAT('创建游戏: ', NEW.name));
END //
DELIMITER ;
```

**触发条件：**

- 事件: INSERT
- 时机: AFTER
- 表: game

**执行逻辑：**

1. 获取新插入记录的 `user_id` 和 `name`
2. 向 `operation_log` 表插入一条记录
3. 操作类型: `GAME_CREATE`
4. 操作目标: `game`
5. 详情: 包含游戏名称

---

**触发条件：**

- 事件: DELETE
- 时机: AFTER

3. 操作类型: `GAME_DELETE`
4. 注意：使用 `OLD` 来获取删除前的值

## 性能考虑

### 触发器性能影响

1. **写入性能**: 触发器会增加INSERT/UPDATE/DELETE的执行时间
2. **日志膨胀**: 每个游戏操作都会生成一条日志记录
3. **索引维护**: 大量日志可能导致索引变大

### 优化建议

1. **异步日志**: 考虑使用消息队列异步记录日志
2. **日志分表**: 按日期对操作日志表进行分区
3. **定期清理**: 使用 `sp_cleanup_old_tasks` 定期清理过期数据
4. **监控触发器**: 定期检查触发器执行情况

## 定时任务配置

### Spring Schedule 配置示例

```java
@Configuration
@EnableScheduling
public class ScheduleConfig {

    @Autowired
    private DataSource dataSource;

    // 每天凌晨2点清理30天前的任务
    @Scheduled(cron = "0 0 2 * * *")
    public void cleanupOldTasks() {
        try (Connection conn = dataSource.getConnection();
             CallableStatement stmt = conn.prepareCall("CALL sp_cleanup_old_tasks(?)")) {
            stmt.setInt(1, 30);
            stmt.execute();
        } catch (SQLException e) {
            log.error("Failed to cleanup old tasks", e);
        }
    }

    // 每天凌晨1点更新游戏统计
    @Scheduled(cron = "0 0 1 * * *")
    public void updateGameStatistics() {
        try (Connection conn = dataSource.getConnection();
             CallableStatement stmt = conn.prepareCall("CALL sp_update_game_statistics()")) {
            stmt.execute();
        } catch (SQLException e) {
            log.error("Failed to update game statistics", e);
        }
    }
}
```
